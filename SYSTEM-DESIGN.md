# LP生成システム 設計思想まとめ

> このドキュメントは、当初の構想と現在の実装方針の経緯・考え方をまとめたものです。

---

## 背景

社内マーケターから「自分たちでLPをつくりたい」という相談を受けたことが起点。  
半年以内に成果を出すためには、素早い制作と効果検証のサイクルが不可欠。  
デザイナーが設計した「型」をベースに、マーケターが自走できる仕組みを目指した。

### マーケターの課題

| # | 課題 |
|---|------|
| 01 | 限られた時間で複数のtoB LPを早く量産したい |
| 02 | デザイナーのリソース不足により内製が難しい |
| 03 | マーケで作りたいが、品質は担保したい |

---

## ① 当初の予定：MD × AI でLP生成

### 構成

3つのmdファイルを用意し、AIがそれを読み込んでHTMLを生成する。

```
mdA：design-system.md（固定）  ─┐
mdB：lp-structure.md（固定）   ─┼─→ Claude AI → HTML
mdC：content-source.md（可変） ─┘
```

| ファイル | 役割 | 変更者 |
|---------|------|--------|
| mdA デザインシステム | カラー・フォント・余白などのビジュアルルール | デザイナー（固定） |
| mdB LP構造 | S1〜S11 全セクションのレイアウト・構造定義 | デザイナー（固定） |
| mdC コンテンツ原稿 | 見出し・本文・CTA・FAQなどの実際のテキスト | マーケター（可変） |

### この方針の課題

- **AIの出力がブレる** — 同じ入力でも毎回違うHTMLが生成されることがあり、デザイン品質を担保できない
- **再生成ループが発生する** — プレビューがないため「生成 → 確認 → 修正 → 再生成」を繰り返す必要があり、非エンジニアには高負荷
- **育成コストがかかる** — AIのブレを矯正するためにSKILL（指示の蓄積）を育てる必要があり、安定するまでに時間がかかる
- **Markdownの編集ハードル** — 非エンジニアにとってmdファイルの直接編集は抵抗が大きい

## ② 現在の実装方針：ウィザード × Claude でLP生成

### 構成

デザインの型はコードに焼き込み、マーケターはGUIで操作するだけでHTMLが生成できる。  
Claudeはウィザードの「パターンを増やす役割」として登場する。

```
デザイナー
  └─ Figmaでデザインシステムを定義
        ↓ Figma MCP で参照
      Claude ←─ マーケターが「こんなパターンが欲しい」とリクエスト
        ↓ デザインルールに沿って生成
      新セクションパターン追加
        ↓ ウィザードが育つ
      LP生成ウィザード（S1〜S11 ブロック）
        ↓ セクション選択 + テキスト入力
        ↓ リアルタイムプレビュー確認
      HTML完成
```

### ウィザードの仕組み（レゴブロック型）

- **S1〜S11** の11セクションが独立したブロックとして設計されている
- 各セクションは **ON/OFF**・**レイアウトパターン選択（0/1/2）**・**カラー選択（A/B/C）** が可能
- マーケターはテキストを入力するだけ。デザインは守られる

### この方針のメリット

| 観点 | 内容 |
|------|------|
| 安定性 | コードがHTMLを生成するため、出力が毎回一致する |
| プレビュー | 入力しながらリアルタイムで見た目を確認できる |
| ハードル低 | GUIフォームなのでマーケターがすぐ使える |
| 育成の方向 | ブレの矯正ではなく「パターンの蓄積」として資産化できる |

### mdファイルの役割（設計思想は継続）

3ファイル構成の **設計思想はそのまま引き継ぐ**。  
ただしAIへの指示ファイルとしてではなく、**コードの設計基盤・ドキュメント** として活用する。

| ファイル | 現在の役割 |
|---------|-----------|
| デザインシステム | `htmlGenerator.ts` のスタイル定義の基盤 |
| LP構造 | S1〜S11セクション設計の基盤 |
| コンテンツ原稿 | ウィザード完了後にダウンロードできるMDファイル（controlの記録） |

## Claudeの接点

当初は「AIがHTMLを生成する」という役割だったが、方針変更後は以下の役割に変わった。

| 当初 | 現在 |
|------|------|
| mdを読んでHTMLを生成する | Figmaのデザインシステムを参照してウィザードの新パターンを生成する |
| マーケターが毎回使う | デザイナー × マーケターが協力してパターンを追加するときに使う |
| 出力がブレる（課題） | デザインルールに拘束されるためブレない |

### 育成の概念の転換

| | 当初の育成（ネガティブ） | 新しい育成（ポジティブ） |
|---|---|---|
| 目的 | AIのブレを矯正する | ウィザードのパターンを増やす |
| 感覚 | 補正・修理 | 積み上げ・資産化 |
| 方向性 | 現状維持のため | 前進するため |

## ③ Figma連携の実装（2026-04-14）

### やったこと

Figma MCP（`use_figma` / `create_new_file` / `get_design_context`）を使い、ウィザードとFigmaを接続する基盤を構築した。

#### 作成したFigmaファイル

- **ファイル名**: LP Systems - Section Components
- **チーム**: YO.Tec
- **ページ構成**: "Layout A" / "Layout B" / "Layout C"（各ページにS1〜S11の全セクション）

#### レイヤー命名規則（コードとFigmaのマッピングキー）

```
s{N}/{fieldName}/{index}
```

例: `s1/logoText/0` → TypeScriptの `data.s1.logoText` に対応。  
この規則を守ることで、sync-from-figma スクリプト（後述）がテキスト値を自動抽出できる。

#### コード側の変更

| ファイル | 変更内容 |
|----------|----------|
| `src/lib/types.ts` | `LayoutIndex = 0 \| 1 \| 2` → `0 \| 1 \| 2 \| 3`（4パターン目に拡張） |
| `src/lib/sectionLayouts.ts` | 型を`LayoutPattern[]`（可変長配列）に変更。S1にパターンD（ミニマル）を追加 |
| `src/components/wizard/LayoutPicker.tsx` | ラベル配列に"D"を追加。グリッド列数をパターン数に応じて動的化 |
| `src/components/editor/EditorPanel.tsx` | `LAYOUT_LABELS`に"D"を追加 |
| `src/components/preview/SectionRenderer.tsx` | S1 layout=3（ミニマル）のReactコンポーネントを追加 |
| `src/lib/htmlGenerator.ts` | S1 layout=3（ミニマル）のHTMLとCSSを追加 |

### 実証したワークフロー

```
マーケター「ヘッダーのミニマルなパターンが欲しい」
  ↓ テキストリクエスト
Claude がFigmaにデザインを作成（use_figma）
  ↓
Claude がコードに反映（6ファイルを変更）
  ↓
ウィザードにパターンDが出現し、プレビューで確認できる
  ↓
HTMLエクスポートも対応
```

## 役割分担の設計

エンジニアが「仕組み」を作り、デザイナー/マーケターが「パターン」を育てる構造を目指す。

### エンジニアの担当範囲（インフラ）

- S1〜S11のセクション型定義（`types.ts`）
- ウィザードUI・プレビュー・HTML出力の基盤
- sync-from-figma / sync-to-figma スクリプト
- CLAUDE.md の整備（非エンジニアがClaudeに依頼するためのルール）

### デザイナー・マーケターの担当範囲（コンテンツ）

- Figmaでパターンのビジュアルデザインを作成
- ClaudeにテキストでパターンリクエストをするとFigmaとコードに自動反映
- デフォルトコンテンツ（`defaultContent.ts`）の文言調整
- ウィザード上でセクションON/OFF・パターン選択・テキスト入力

### CLAUDE.md の役割

非エンジニアがClaudeを使うための取扱説明書。以下のルールを記述予定:

```
- 新パターン追加時に変更すべき4ファイルのリスト
- ブロック座標の書き方（sectionLayouts.ts の blocks 配列形式）
- SVGサムネイルの描画ルール
- SectionRenderer.tsx と htmlGenerator.ts の対応追加方法
```
